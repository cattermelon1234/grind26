## Blocked Dot Product in Cuda
We want to assign each thread to compute a partial product of two indices, and somehow reduce
the partial products to get the final dot product. 

The simple plan is: use shared memory to store the partial products, and then use a reduction to sum them up.

1. Each thread computes a portion of the dot product and accumulates the result into a local variable. If the vector is larger than the total number of threads launched, a single thread may be responsible for multiple vector elements. To accomplish this, we use a **grid-stride loop**.

First, each thread computes its global index:

```cpp
int tid = blockIdx.x * blockDim.x + threadIdx.x;
```

The total number of threads in the grid is:

```cpp
int stride = blockDim.x * gridDim.x;
```

Thread `tid` processes indices:

```text
tid
tid + stride
tid + 2 * stride
...
```

until it exceeds the vector length.

This is sometimes necessary in production patterns where we do not want the number of threads to scale with vector size.

2. Shared Memory 
In order for reduction to happen, each thread needs to be aware of the products computed 
by other threads in the same block. We can use shared memory for this. Each thread will write its partial 
product to a shared memory array, declared with the __shared__ keyword. This allows all 
threads in the block to access the partial products.

__shared__ float partialProducts[blockDim.x];

Shared memory is a shared space on each SM (streaming multiprocessor) that can be accessed by 
all threads in a block. Shared memory is much faster than global memory, so we want to use it 
for intermediate results. However, it is limited in size, so we need to ensure that our block 
size does not exceed the available shared memory.

3. Reduction 
Typically, we use a parallel reduction technique to sum up the partial products. This involves 
iteratively halving the number of active threads and summing pairs of elements in the shared 
memory array until only one thread remains, which will hold the final dot product for that block.

At first glance, it seems like the below approach is intuitive:
We combine neighboring threads to sum their products, and keep halving like discussed. This leads 
to a log number of steps. 
In this case, a 'stride' is used to determine which threads are active together. Initially, the 
stride is 1, meaning each thread sums with its neighbor. Then the stride doubles to 2, meaning 
each thread sums with the thread that is 2 away, and so on.

          56
       20    36
    6     14    22
 1     5     9     13
0 1   2 3   4 5   6  7   

However, this approach can lead to thread divergence, where some threads are active while others are idle.
Initially, threads 0, 2, 4, 6 are active, then threads 0, 4 are active, and finally thread 0 is active. 
This can lead to inefficient use of the GPU's resources.

We prefer if neighboring threads are active together, and the reason is because: GPUs execute warps, not individual
threads. If our threads are sparse, then we may have to execute multiple warps with very low occupancy, which 
can lead to underutilization of the GPU. 

This can be achieved by a different reduction pattern.

          56
       24    32
    10    14    18
 4     6     8     10
0 4   1 5   2 6   3  7

In this pattern, threads 0, 1, 2, 3 are active together, then threads 0, 1 are active together, and finally thread 0 is active.
It may be difficult to see, but the difference is just each thread is summing with a faraway 
thread, instead of a neighboring thread. This way, we can ensure that all threads in a warp are
active together, leading to better performance.

In this reduction pattern, a stride determines how far away the value being added is in shared memory. Initially, the stride is half the block size, so thread 0 adds the value from thread stride, thread 1 adds the value from thread stride + 1, and so on. After each iteration, the stride is halved, reducing the number of active threads by half while keeping the active threads contiguous. This continues until only thread 0 remains, which contains the sum for the entire block.

4. Final Result, modifying Global Memory
Once this reduction is achieved, we will have the final dot product for that block. If we have multiple blocks,
we can use atomic operations to sum the results from each block into a global variable, or we can launch 
a second kernel to perform the reduction on the block results.

Global memory is much slower than shared memory, so we want to minimize the number of accesses to global memory.
